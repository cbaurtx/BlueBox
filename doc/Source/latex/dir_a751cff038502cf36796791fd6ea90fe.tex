\doxysubsection{Detailed Description}
\hypertarget{dir_a751cff038502cf36796791fd6ea90fe_autotoc_md213}{}\doxysubsection{\texorpdfstring{llhttp}{llhttp}}\label{dir_a751cff038502cf36796791fd6ea90fe_autotoc_md213}
\href{https://github.com/nodejs/llhttp/actions?query=workflow\%3ACI}{\texttt{}}

Port of \href{https://github.com/nodejs/http-parser}{\texttt{http\+\_\+parser}} to \href{https://github.com/nodejs/llparse}{\texttt{llparse}}.\hypertarget{dir_a751cff038502cf36796791fd6ea90fe_autotoc_md214}{}\doxysubsection{\texorpdfstring{Why?}{Why?}}\label{dir_a751cff038502cf36796791fd6ea90fe_autotoc_md214}
Let\textquotesingle{}s face it, \href{https://github.com/nodejs/http-parser}{\texttt{http\+\_\+parser}} is practically unmaintainable. Even introduction of a single new method results in a significant code churn.

This project aims to\+:


\begin{DoxyItemize}
\item Make it maintainable
\item Verifiable
\item Improving benchmarks where possible
\end{DoxyItemize}

More details in \href{https://youtu.be/x3k_5Mi66sY}{\texttt{Fedor Indutny\textquotesingle{}s talk at JSConf EU 2019}}\hypertarget{dir_a751cff038502cf36796791fd6ea90fe_autotoc_md215}{}\doxysubsection{\texorpdfstring{How?}{How?}}\label{dir_a751cff038502cf36796791fd6ea90fe_autotoc_md215}
Over time, different approaches for improving \href{https://github.com/nodejs/http-parser}{\texttt{http\+\_\+parser}}\textquotesingle{}s code base were tried. However, all of them failed due to resulting significant performance degradation.

This project is a port of \href{https://github.com/nodejs/http-parser}{\texttt{http\+\_\+parser}} to Type\+Script. \href{https://github.com/nodejs/llparse}{\texttt{llparse}} is used to generate the output C source file, which could be compiled and linked with the embedder\textquotesingle{}s program (like \href{https://github.com/nodejs/node}{\texttt{Node.\+js}}).\hypertarget{dir_a751cff038502cf36796791fd6ea90fe_autotoc_md216}{}\doxysubsection{\texorpdfstring{Performance}{Performance}}\label{dir_a751cff038502cf36796791fd6ea90fe_autotoc_md216}

llhttp is faster by approximately {\bfseries{156\%}}.\hypertarget{dir_a751cff038502cf36796791fd6ea90fe_autotoc_md217}{}\doxysubsection{\texorpdfstring{Maintenance}{Maintenance}}\label{dir_a751cff038502cf36796791fd6ea90fe_autotoc_md217}
llhttp project has about 1400 lines of Type\+Script code describing the parser itself and around 450 lines of C code and headers providing the helper methods. The whole \href{https://github.com/nodejs/http-parser}{\texttt{http\+\_\+parser}} is implemented in approximately 2500 lines of C, and 436 lines of headers.

All optimizations and multi-\/character matching in llhttp are generated automatically, and thus doesn\textquotesingle{}t add any extra maintenance cost. On the contrary, most of \doxylink{structhttp__parser}{http\+\_\+parser}\textquotesingle{}s code is hand-\/optimized and unrolled. Instead describing "{}how"{} it should parse the HTTP requests/responses, a maintainer should implement the new features in \href{https://github.com/nodejs/http-parser}{\texttt{http\+\_\+parser}} cautiously, considering possible performance degradation and manually optimizing the new code.\hypertarget{dir_a751cff038502cf36796791fd6ea90fe_autotoc_md218}{}\doxysubsection{\texorpdfstring{Verification}{Verification}}\label{dir_a751cff038502cf36796791fd6ea90fe_autotoc_md218}
The state machine graph is encoded explicitly in llhttp. The \href{https://github.com/nodejs/llparse}{\texttt{llparse}} automatically checks the graph for absence of loops and correct reporting of the input ranges (spans) like header names and values. In the future, additional checks could be performed to get even stricter verification of the llhttp.\hypertarget{dir_a751cff038502cf36796791fd6ea90fe_autotoc_md219}{}\doxysubsection{\texorpdfstring{Usage}{Usage}}\label{dir_a751cff038502cf36796791fd6ea90fe_autotoc_md219}

The settings object contains a list of callbacks that the parser will invoke.

The following callbacks can return {\ttfamily 0} (proceed normally), {\ttfamily -\/1} (error) or {\ttfamily HPE\+\_\+\+PAUSED} (pause the parser)\+:


\begin{DoxyItemize}
\item {\ttfamily on\+\_\+message\+\_\+begin}\+: Invoked when a new request/response starts.
\item {\ttfamily on\+\_\+message\+\_\+complete}\+: Invoked when a request/response has been completedly parsed.
\item {\ttfamily on\+\_\+url\+\_\+complete}\+: Invoked after the URL has been parsed.
\item {\ttfamily on\+\_\+method\+\_\+complete}\+: Invoked after the HTTP method has been parsed.
\item {\ttfamily on\+\_\+version\+\_\+complete}\+: Invoked after the HTTP version has been parsed.
\item {\ttfamily on\+\_\+status\+\_\+complete}\+: Invoked after the status code has been parsed.
\item {\ttfamily on\+\_\+header\+\_\+field\+\_\+complete}\+: Invoked after a header name has been parsed.
\item {\ttfamily on\+\_\+header\+\_\+value\+\_\+complete}\+: Invoked after a header value has been parsed.
\item {\ttfamily on\+\_\+chunk\+\_\+header}\+: Invoked after a new chunk is started. The current chunk length is stored in {\ttfamily parser-\/\texorpdfstring{$>$}{>}content\+\_\+length}.
\item {\ttfamily on\+\_\+chunk\+\_\+extension\+\_\+name\+\_\+complete}\+: Invoked after a chunk extension name is started.
\item {\ttfamily on\+\_\+chunk\+\_\+extension\+\_\+value\+\_\+complete}\+: Invoked after a chunk extension value is started.
\item {\ttfamily on\+\_\+chunk\+\_\+complete}\+: Invoked after a new chunk is received.
\item {\ttfamily on\+\_\+reset}\+: Invoked after {\ttfamily on\+\_\+message\+\_\+complete} and before {\ttfamily on\+\_\+message\+\_\+begin} when a new message is received on the same parser. This is not invoked for the first message of the parser.
\end{DoxyItemize}

The following callbacks can return {\ttfamily 0} (proceed normally), {\ttfamily -\/1} (error) or {\ttfamily HPE\+\_\+\+USER} (error from the callback)\+:


\begin{DoxyItemize}
\item {\ttfamily on\+\_\+url}\+: Invoked when another character of the URL is received.
\item {\ttfamily on\+\_\+status}\+: Invoked when another character of the status is received.
\item {\ttfamily on\+\_\+method}\+: Invoked when another character of the method is received. When parser is created with {\ttfamily HTTP\+\_\+\+BOTH} and the input is a response, this also invoked for the sequence {\ttfamily HTTP/} of the first message.
\item {\ttfamily on\+\_\+version}\+: Invoked when another character of the version is received.
\item {\ttfamily on\+\_\+header\+\_\+field}\+: Invoked when another character of a header name is received.
\item {\ttfamily on\+\_\+header\+\_\+value}\+: Invoked when another character of a header value is received.
\item {\ttfamily on\+\_\+chunk\+\_\+extension\+\_\+name}\+: Invoked when another character of a chunk extension name is received.
\item {\ttfamily on\+\_\+chunk\+\_\+extension\+\_\+value}\+: Invoked when another character of a extension value is received.
\end{DoxyItemize}
